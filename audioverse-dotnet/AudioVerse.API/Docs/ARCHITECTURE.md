## Overview

AudioVerse is a karaoke party management platform built on:

- **.NET 10** with ASP.NET Core
- **PostgreSQL** database with EF Core 9
- **Redis** for caching and real-time state
- **MinIO/S3** for file storage
- **SignalR** for real-time communication

## Layer Responsibilities

### API Layer (`AudioVerse.API`)

- HTTP request handling
- Input validation (model binding)
- Authentication/Authorization
- Response serialization
- SignalR real-time communication
- Middleware pipeline

### Application Layer (`AudioVerse.Application`)

- CQRS command/query handling
- Business logic orchestration
- Domain service coordination
- Cross-cutting concerns (logging, validation)
- DTO mapping

### Domain Layer (`AudioVerse.Domain`)

- Entity definitions
- Business rules and invariants
- Repository interfaces (abstractions)
- Domain events (future)

### Infrastructure Layer (`AudioVerse.Infrastructure`)

- Database access (EF Core, Dapper)
- External API integrations
- File storage (MinIO/S3)
- Email services
- Caching (Redis)

## Key Design Patterns

### CQRS with MediatR

```csharp
// Command
public record CreateEventCommand(Event Event) : IRequest<int>;

// Handler
public class CreateEventHandler : IRequestHandler<CreateEventCommand, int>
{
    public async Task<int> Handle(CreateEventCommand request, CancellationToken ct)
    {
        // Business logic
        return await _repository.CreateEventAsync(request.Event);
    }
}

// Usage in Controller
[HttpPost]
public async Task<IActionResult> Create([FromBody] Event ev)
    => Ok(await _mediator.Send(new CreateEventCommand(ev)));
```

### Repository Pattern

- **EF Repository**: Complex queries with eager loading
- **Dapper Repository**: Read-heavy, performance-critical queries

```csharp
// Interface in Domain
public interface IEfKaraokeRepository
{
    Task<KaraokeEvent?> GetEventByIdAsync(int id);
}

// Implementation in Infrastructure
public class KaraokeRepositoryEF : IEfKaraokeRepository
{
    public async Task<KaraokeEvent?> GetEventByIdAsync(int id)
        => await _dbContext.KaraokeParties
            .Include(p => p.EventPlayers)
            .FirstOrDefaultAsync(p => p.Id == id);
}
```

### Options Pattern

```csharp
// Configuration class
public class SpotifyServiceOptions
{
    public string ClientId { get; set; }
    public string ClientSecret { get; set; }
}

// Registration
builder.Services.Configure<SpotifyServiceOptions>(
    builder.Configuration.GetSection("Spotify"));

// Usage
public class SpotifyService
{
    public SpotifyService(IOptions<SpotifyServiceOptions> options)
    {
        _clientId = options.Value.ClientId;
    }
}
```

## Security Architecture

### Authentication Flow

```
Client ? Login Request ? API
                          ?
                          ?
                   Validate Credentials
                          ?
                          ?
                   Generate JWT Token
                   (id, username, roles, iat, exp)
                          ?
                          ?
Client ? JWT Token ? API
```

### Middleware Pipeline

```
Request ? JwtMiddleware ? UserBanMiddleware ? ProfanityMiddleware
              ?                   ?                    ?
              ?                   ?                    ?
        Validate Token    Check Active Bans    Scan for Profanity
              ?                   ?                    ?
              ?                   ?                    ?
        Set User Principal   403 Forbidden      400 Bad Request
                                  ?                    ?
                            (if banned)         (if profanity)
                                  ?                    ?
                                  ??????????????????????
                                           ?
                                           ?
                                    Controller Action
```

### Authorization

- **JWT Bearer** tokens with claims
- **Role-based** access (Admin, User, Guest)
- **Resource-based** checks (e.g., event organizer)

## Real-time Features

### SignalR Hubs

| Hub | Path | Purpose |
|-----|------|---------|
| KaraokeHub | `/hubs/karaoke` | Event updates, scores |
| AdminHub | `/hubs/admin` | Logs, audit, config |
| ModerationHub | `/hubs/moderation` | Reports |

### Lobby State

- **Redis** (preferred): Distributed caching
- **InMemory** (fallback): Single-instance development

```csharp
// Redis connection with retry
if (!string.IsNullOrEmpty(redisConn))
{
    services.AddSingleton<ILobbyStore, RedisLobbyStore>();
}
else
{
    services.AddSingleton<ILobbyStore, InMemoryLobbyStore>();
}
```

### Upload Flow

```csharp
// 1. Request presigned URL
var uploadUrl = await _fileStorage.GetUploadUrlAsync(bucket, key, contentType);

// 2. Client uploads directly to MinIO
// PUT {uploadUrl} with file content

// 3. Confirm upload in API
await _mediator.Send(new ConfirmUploadCommand(bucket, key));
```

## Deployment

### Docker Compose (Development)

```yaml
services:
  api:
    build: .
    ports:
      - "5000:5000"
    depends_on:
      - postgres
      - redis

    image: postgres:16
    environment:
      POSTGRES_DB: audioverse

    image: redis:7-alpine

    image: minio/minio
```

## Performance Considerations

1. **Query Optimization**: Use Dapper for read-heavy endpoints
2. **Caching**: Redis for frequently accessed data
3. **Connection Pooling**: Configure Kestrel limits
4. **Async/Await**: All I/O operations are async
5. **Pagination**: Always paginate list endpoints
