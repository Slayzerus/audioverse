# Rozdział 3 — Projekt i architektura systemu

## 3.1. Wymagania funkcjonalne i niefunkcjonalne

System AudioVerse był projektowany na podstawie zestawu wymagań
zebranych w dwóch kategoriach (tab. 3.1).

**Tabela 3.1. Wymagania funkcjonalne (WF)**

| ID   | Treść wymagania                                                           |
|------|---------------------------------------------------------------------------|
| WF01 | Użytkownik może się zarejestrować i zalogować (Identity Server, JWT)      |
| WF02 | Biblioteka piosenek: przeglądanie, filtrowanie po tytule/artyście/gatunku|
| WF03 | Import piosenki w formacie UltraStar TXT wraz z plikiem audio             |
| WF04 | Tworzenie sesji (party) karaoke z kodem dołączenia                        |
| WF05 | Tryb gry: wizualizacja nut, detekcja pitch w czasie rzeczywistym          |
| WF06 | Ocena śpiewu per nuta (perfect/good/miss) i Score 0–10 000 pkt           |
| WF07 | Wyświetlenie rankingu po zakończonej rundzie                               |
| WF08 | Moduł laboratoryjny: pomiary algorytmów detekcji F0 i generowanie PDF     |

**Wymagania niefunkcjonalne (WNF):**

- WNF01: Latencja detekcji pitch (pYIN real-time) < 150 ms od wysłania PCM.
- WNF02: System działa w przeglądarce bez konieczności instalowania wtyczek.
- WNF03: Aplikacja jest wdrażana jedną komendą (`docker compose up`).
- WNF04: Użytkownik może korzystać z systemu na ekranach ≥ 1280 × 720 px.
- WNF05: Raporty PDF z modułu laboratoryjnego generowane w < 5 s.


## 3.2. Architektura systemu

System AudioVerse zbudowany jest w oparciu o architekturę mikroserwisową
złożoną z czterech warstw logicznych komunikujących się przez REST, WebSocket
i SignalR. Pełna architektura systemu z zależnościami między warstwami
przedstawiona jest na rysunku 3.1.

> **Rysunek 3.1** — Schemat komunikacji między warstwami systemu AudioVerse:
> przeglądarka → ASP.NET API → mikroserwisy AI (FastAPI) → PostgreSQL / MinIO.
> Dla każdego mikroserwisu podano protokół (HTTP/WebSocket) i średnią latencję.
> Źródło: własne, diagram dostępny w pliku
> `AudioVerse.API/Docs/diagrams/01-system-architecture.drawio`.

Frontend przesyła strumień PCM przez surowe WebSocket do mikroserwisu
`audio_pitch`. Wyniki F0 wracają do frontendu, gdzie silnik karaoke
(`karaokeTimelineRenderer.ts`) nanosi kursor pitch na wizualizację nut.
Backend `.NET` pośredniczy w synchronizacji stanu sesji przez SignalR
i zapisuje wyniki gry do PostgreSQL.


## 3.3. Model danych

Moduł karaoke oparty jest na siedmiu encjach głównych (tab. 3.2, rys. 3.2):

**Tabela 3.2. Encje domeny karaoke**

| Encja                     | Kluczowe pola                                             |
|---------------------------|-----------------------------------------------------------|
| KaraokeSong               | Id, Title, Artist, Language, Genre, CoverImagePath       |
| KaraokeSongFile           | Id, SongId, S3Key, Format (UltraStar/MP3), BPM, GAP      |
| KaraokeSession            | Id, Code (join code), HostId, Status, CreatedAt          |
| KaraokeSessionRound       | Id, SessionId, SongFileId, Status, StartedAt, EndedAt    |
| KaraokeSessionRoundPlayer | Id, RoundId, PlayerId, Slot (1–4)                        |
| KaraokeSinging            | Id, RoundId, RoundPartId, PlayerId, **Score, Hits, Misses, Good, Perfect, Combo** |
| KaraokeSingingRecording   | Id, SingingId, S3Key, DurationMs                         |

> **Rysunek 3.2** — Diagram encji modelu danych modułu karaoke (ER).
> Widoczne relacje między KaraokeSession, KaraokeSessionRound,
> KaraokeSessionRoundPlayer i KaraokeSinging.
> Źródło: własne, wygenerowany diagram dostępny w pliku
> `AudioVerse.API/Docs/diagrams/auto-data-model.drawio`,
> wygenerowany narzędziem `AudioVerse.DiagramGenerator`.

Pole `Score` w `KaraokeSinging` przechowuje skumulowane punkty gracza
(0–10 000). `Hits`, `Misses`, `Good`, `Perfect` to liczniki wynikające
z oceny per nuta. `Combo` to maksymalna seria kolejnych trafionych nut
uzyskana w trakcie rundy.


## 3.4. Stos technologiczny

System AudioVerse został zbudowany w oparciu o technologie dedykowane
poszczególnym warstwom: backend w ASP.NET Core/.NET 10 (C#), mikroserwisy AI
w Python 3.11 z FastAPI, frontend w React 18 + TypeScript. Pełne zestawienie
wyborów technologicznych wraz z uzasadnieniem zawiera tabela 3.3.
Szczegółowy schemat komunikacji między warstwami i zależności serwisów
przedstawiono w podrozdziale 3.2 (rys. 3.1); implementacja komunikacji
omówiona jest szczegółowo w rozdziale 4.

**Tabela 3.3. Stos technologiczny AudioVerse**

| Warstwa             | Technologia                        | Uzasadnienie wyboru                 |
|---------------------|------------------------------------|-------------------------------------|
| Backend API         | ASP.NET Core 10, C# 13             | Typowanie, CQRS, wbudowany SignalR  |

| CQRS/Bus            | MediatR 12                         | Separacja komend i zapytań          |

## 3.5. Komunikacja w czasie rzeczywistym — ogólna koncepcja

W sesji karaoke wymagana jest dwukierunkowa, niskolatencyjna komunikacja
między przeglądarką a serwerem. W systemie AudioVerse zastosowano dwa kanały:
protokół SignalR do synchronizacji stanu sesji oraz surowe WebSocket
do strumieniowania PCM z mikrofonu. Dzięki temu możliwe jest działanie
detekcji pitch w czasie rzeczywistym z latencją poniżej 150 ms.
Szczegółowa implementacja obu kanałów opisana jest w rozdziale 4.
