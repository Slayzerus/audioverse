# Rozdział 4 — Implementacja

## 4.1. Backend — AudioVerse.API (.NET 10)

Backend zorganizowany jest według wzorca CQRS zaimplementowanego
przez bibliotekę MediatR 12 (rys. 4.1). Każda operacja klienta transluje się na
komendę (`Command`) lub zapytanie (`Query`), obsługiwane przez dedykowane
`Handler` w warstwie `AudioVerse.Application`. Kontrolery są cienkie —
zawierają jedynie akcję HTTP i delegację do mediatora:

```csharp
[HttpPost("singing")]
public async Task<IActionResult> SaveSinging(
    [FromBody] SaveKaraokeSingingCommand cmd,
    CancellationToken ct)
    => Ok(await _mediator.Send(cmd, ct));
```

> **Rysunek 4.1** — Diagram architektury CQRS w AudioVerse: Controller → MediatR
> → Handler → Repository → DbContext → PostgreSQL. Widoczne przykłady:
> SaveKaraokeSingingCommand → SaveKaraokeSingingHandler (zapis wyników) i
> GetRankingsQuery → GetRankingsHandler (odczyt rankingów).
> Źródło: własne, diagram dostępny w pliku
> `AudioVerse.API/Docs/diagrams/08-cqrs-architecture.drawio`.

### 4.1.1. Zarządzanie sesjami i rundami

`KaraokeController` (trasa `/api/karaoke`) udostępnia operacje (rys. 4.2):

### 4.1.2. Baza piosenek i format UltraStar

`KaraokeSongsController` obsługuje CRUD piosenek oraz wersje pliku
(jedna piosenka może mieć kilka wersji dostosowanych do różnych

(freestyle), `-` (pauza), `#` (rap), `G` (rap-złota).

### 4.1.3. Komunikacja w czasie rzeczywistym — KaraokeHub (SignalR)

`KaraokeHub` nasłuchuje zdarzeń (rys. 4.3):
- `StartRound(roundId)` — emitowany do wszystkich graczy,
- `ScoreUpdate(playerId, score, combo)` — aktualizacja co 5 s,
- `RoundFinished(rankings)` — wyniki po zakończeniu piosenki.

Frontend rejestruje handlery przez SDK SignalR TypeScript:

```typescript
connection.on("RoundFinished", (rankings: PlayerRanking[]) => {
    navigate(`/karaoke/ranking/${roundId}`);
});
```

Oprócz SignalR system korzysta z surowego kanału WebSocket (trasa `/ws/pitch`)
do strumieniowania PCM 16-bit, mono, 16 kHz z przeglądarki. Bufor
`Int16Array` (4096 próbek, 256 ms) przesyłany jest przez `WebSocket.send()`;
serwer Python akumuluje sygnał do 1 sekundy i odsyała wynik F0 jako JSON
`{hz, confidence}`. Szczegóły implementacji WebSocket opisano
w sekcji 4.2.1.

## 4.2. Mikroserwisy AI (Python / FastAPI)

Każdy mikroserwis uruchomiony jest w osobnym kontenerze Dockera
z bazowym obrazem `python:3.11-slim`. Rejestrują metryki latencji do
Prometheusa przez `prometheus_fastapi_instrumentator`.

### 4.2.1. audio_pitch — detekcja wysokości dźwięku

Mikroserwis `audio_pitch` (port 8001) obsługuje:
- `POST /pitch` — wgraj plik WAV/MP3, resampling librosa do 16 kHz mono,
  uruchomienie CREPE lub pYIN (parametr `algorithm`), zwróć trajektorię.
- `WebSocket /ws/pitch_server` — strumieniowanie PCM 16-bit 16 kHz;
  bufor akumulowany do 1 s, pYIN uruchamiane per bufor, F0 zwracane jako JSON.

```python
@app.websocket("/ws/pitch_server")
async def ws_pitch(websocket: WebSocket):
    await websocket.accept()
    buffer = bytearray()
    while True:
        data = await websocket.receive_bytes()
        buffer.extend(data)
        if len(buffer) >= SR * 2:          # 1 sekunda PCM 16-bit
            audio = np.frombuffer(buffer, dtype=np.int16).astype(np.float32) / 32768
            f0, _, probs = librosa.pyin(audio, fmin=80, fmax=1000, sr=SR)
            hz  = float(np.nanmedian(f0))
            conf = float(np.nanmean(probs))
            await websocket.send_json({"hz": hz, "confidence": conf})
            buffer.clear()
```

### 4.2.2. sing_score — ocena śpiewu metodą DTW

Mikroserwis `sing_score` (port 8002):
- `POST /score` — przyjmuje dwa pliki WAV (śpiewak + referencja),
  wyznacza trajektorie F0 obu (CREPE lub pYIN), normalizuje do
  wspólnego grid czasowego, oblicza dystans DTW (dtaidistance),
  mapuje na score 0–100.
- `WebSocket /score/live` — streaming score co 10 ramek PCM.

Normalizacja score: $s = 100 \cdot (1 - d / d_{\max})$, gdzie $d_{\max}$
to empiryczny percentyl 99 ze zbioru kalibracyjnego.

## 4.4. Silnik karaoke — timeline i detekcja pitch w przeglądarce

Silnik gry karaoke zawarty jest w dwóch modułach TypeScript:

- `karaokeNoteParsing.ts` — parsowanie pliku UltraStar TXT,
- `karaokeTimelineRenderer.ts` — renderowanie na Canvas 2D.

### 4.4.1. Parsowanie pliku UltraStar po stronie klienta

Funkcja `parseNotes()` przetwarza tablicę linii pliku TXT na obiekty
`KaraokeNoteData`:

```typescript
interface KaraokeNoteData {
    type:       ':' | '*' | 'F' | '#' | 'G';
    startTime:  number;   // sekundy od początku audio
    endTime:    number;
    pitch:      number;   // UltraStar pitch (0–11 + oktawa)
    text:       string;   // sylaba
}
```

BPM pobierany jest z nagłówka `#BPM:` obsługującego zarówno separator
przecinkowy (`148,50`) jak i kropkowy (`148.50`). Jeśli nagłówek jest
nieobecny, stosowany jest fallback 120 BPM.

Przeliczanie beat → sekundy według formuły UltraStar WorldParty
(zob. sekcja 2.12):

```typescript
const startTime = (beat * 15) / bpm + gap / 1000;
const endTime   = ((beat + duration) * 15) / bpm + gap / 1000;
```

### 4.4.2. Konwersja Hz na pitch UltraStar — octave folding

Funkcja `hzToUltrastarPitch(hz, targetNotePitch)` implementuje
algorytm octave folding z oryginalnego kodu UltraStar:

- $\text{midi} = \text{round}(69 + 12 \log_2(hz / 440))$
- $\text{detected} = \text{midi} \bmod 12$ (pitch class 0–11)
- Pętla: dopóki $\text{detected} - \text{target} > 6$: $\text{detected} -= 12$
- Pętla: dopóki $\text{detected} - \text{target} < -6$: $\text{detected} += 12$

Wynik to pitch w oktawie najbliższej nucie docelowej, co eliminuje
błędy oktawowe przy ocenie śpiewu.

### 4.4.3. Renderowanie nut i efekty wizualne

`karaokeTimelineRenderer.ts` (893 linie) rysuje na Canvas 2D:

- **Paski nut** — zaokrąglone prostokąty (cap style gloss) w kolorze
  zależnym od gracza; złote nuty (`*`) rysowane złotą gradientem.
- **Kursor pitch** — pionowa linia nakładana na timeline z aktualnie
  wykrytą przez pYIN wysokością dźwięku.
- **Efekt cząsteczkowy** dla nut zlatych — animacja `BouncingBall`
  emitująca 12 iskier przy trafieniu nuty golden.
- **Combo display** — licznik w prawym górnym rogu, animowany skalą
  przy wzroście combo.
- **Verse ratings** — pasek z oceną wersetową (Perfect / Great / Good /
  Miss) po zakończeniu każdej frazy.

### 4.4.4. System punktacji i combo

Per każdą ramkę (co 16 ms) funkcja `scoreFrame()` oblicza:

- Wykryty pitch `hzToUltrastarPitch(f0, noteTarget)`.
- `error = |detected - noteTarget|` — błąd półtonowy.
- Klasyfikacja: `error == 0` → Perfect; `error ≤ 1` → Good; inaczej Miss.
- Przyrost score: Perfect → +10 pkt × mnożnik; Good → +5 pkt; Miss → 0.
- Combo: rośnie przy Perfect/Good, zeruje się przy Miss.
- Złota nuta (`*`): punkty pomnożone × 2.

Wynik zapisywany jest w encji `KaraokeSinging` przez endpoint
`POST /api/karaoke/singing`.


## 4.5. Ocena śpiewu — kryteria i metryki

System AudioVerse stosuje wielowymiarowy model oceny:

- **Score** (0–10 000 pkt) — skumulowana punktacja za każdą nutę,
  przeliczana ze wskaźnika DTW przez normę liniową.
- **Hits / Misses / Good / Perfect** — kumulatywne liczniki nut,
  przypisane w locie na podstawie cent-error.
- **Combo** — seria kolejnych nut z cent-error < 60c.
- **Golden notes** — nuty oznaczone `*` w pliku UltraStar dają podwojoną
  punktację.

**Metryki eksperymentalne** (moduł laboratoryjny):

- **RMSE Hz**: $\sqrt{\frac{1}{N}\sum_i (f_i^{\text{est}} - f_i^{\text{ref}})^2}$
- **RMSE cents**: $\frac{1200}{\ln 2} \cdot \sqrt{\frac{1}{N}\sum_i \ln^2\!\left(\frac{f_i^{\text{est}}}{f_i^{\text{ref}}}\right)}$
- **Dokładność Acc@50c**: procent ramek z $|\text{error}| < 50c$
- **Pearson r**: korelacja liniowa trajektorii F0

RMSE w centach (a nie Hz) jest właściwą metryką dla muzyki, gdyż skala
chromatyczna jest logarytmiczna: odległość pół tonu = 100 centów, niezależnie
od rejestru.

```yaml
services:

  audioverse-identity:   # Identity Server, port 5001
  audio-pitch:           # FastAPI pYIN+CREPE, port 8001
  sing-score:            # FastAPI DTW, port 8002
  audio-rhythm:          # FastAPI rhythm, port 8004
  audio-vad:             # FastAPI VAD, port 8005
  audio-analysis:        # FastAPI analiza całościowa, port 8006
  postgresql:            # PostgreSQL 16, vol. pgdata
  minio:                 # MinIO S3, porty 9000/9001
  redis:                 # Redis Stack, port 6379
```

Serwisy AI z GPU (`audio-pitch`) mają sekcję:

```yaml
deploy:
  resources:
    reservations:
      devices:
        - driver: nvidia
          count: 1
          capabilities: [gpu]
```

Uruchomienie systemu wymaga jednej komendy: `docker compose up -d`.
Migracje bazy danych uruchamiane są automatycznie podczas startu
`audioverse-api` przez `context.Database.MigrateAsync()`.

Konfiguracja środowiska GPU (sterowniki NVIDIA, CUDA, odpowiednia wersja
biblieki PyTorch/torchcrepe) może się różnić w zależności od modelu karty
graficznej. W celu uproszczenia instalacji powstał dedykowany kreator
**AudioVerse SetupWizard** (projekt `AudioVerse.SetupWizard` w repozytorium
`audioverse-dotnet`, gotowy plik wykonywalny w katalogu `Instalator`).
Kreator przeprowadza użytkownika przez rozpoznanie sprzętu, dobór
